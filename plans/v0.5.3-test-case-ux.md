# Plan: v0.5.3 — Test Case UX Improvements

**Status:** Design-reviewed
**Branch:** master
**Date:** 2026-04-10
**Design score:** 4/10 → 9/10

## Overview

Three UX improvements to the test case management workflow:

1. **Tag filtering on Test Cases tab** — same tag-chip filter as test suite search/manage dialogs, added to the Cases tab and Run tab.
2. **Import batch visibility** — newly imported test cases are visually distinguishable with amber left-border + IMPORTED badge; stacked per-batch banners for undo/cherry-pick; persistent via schema change.
3. **JIRA key format enforcement** — auto-detect and correct malformed JIRA keys during import; show correction preview for user approval; inline form shows correction hint vs. error with distinct styles.

```ts
const filteredCases = useMemo(() => {
  let result = testCases;
  if (caseSearch.trim()) {
    const q = caseSearch.toLowerCase();
    result = result.filter(tc =>
      tc.title.toLowerCase().includes(q) ||
      tc.displayId.toLowerCase().includes(q) ||
      (tc.module ?? "").toLowerCase().includes(q) ||
      (tc.jiraKey ?? "").toLowerCase().includes(q) ||
      (tc.tags ?? []).some(t => t.toLowerCase().includes(q))
    );
  }
  if (caseTagFilter.length > 0) {
    result = result.filter(tc =>
      caseTagFilter.some(tag => (tc.tags ?? []).includes(tag))
    );
  }
  return result;
}, [testCases, caseSearch, caseTagFilter]);
```

## Feature 2: Import Batch Visibility

### Schema change (DECIDED: persistent)
Add `importBatchId` to `TestCase` with proper Prisma FK relation (DECIDED: onDelete: SetNull):
```prisma
model TestCase {
  // ... existing fields ...
  importBatchId String?
  importBatch   ImportBatch? @relation(fields: [importBatchId], references: [id], onDelete: SetNull)
}
```

Add `ImportBatch` tracking model:
```prisma
model ImportBatch {
  id          String     @id @default(cuid())
  projectId   String
  filename    String
  importedAt  DateTime   @default(now())
  // NOTE: no caseCount — count is computed live from TestCase.importBatchId references
  dismissed   Boolean    @default(false)
  project     Project    @relation(fields: [projectId], references: [id], onDelete: Cascade)
  cases       TestCase[]

  @@index([projectId])
  @@map("import_batches")
}
```

`dismissed = true` clears badge rendering for all users (project-scoped dismissal is intentional).

When a test case is added to any suite, clear its `importBatchId` server-side. This applies to:
- `POST /api/test-suites/[id]/cases` (internal)
- `POST /api/v1/test-suites/[id]/cases` (public v1 API) — ADDED per Codex review

Wrap `createMany` + `updateMany({ importBatchId: null })` in a `$transaction`.

### Import API changes
- Generate a `batchId = cuid()` at import start
- Create an `ImportBatch` record before inserting cases
- Set `importBatchId` on each created `TestCase`
- Return `batchId` and `filename` in the API response alongside `created` count
- Also fetch and return active batches so the frontend can refresh banners

### Per-batch stacked banners (DECIDED: stack, not merge)

Multiple banners, one per undismissed batch, rendered above the table. Newest on top.

```
┌───────────────────────────────────────────────────────────┐
│ ✔ 15 cases imported from regression-suite.xlsx        ×  │
│ [Undo all]  [Select imported]                             │
└───────────────────────────────────────────────────────────┘
┌───────────────────────────────────────────────────────────┐
│ ✔ 47 cases imported from acme-tests.xlsx              ×  │
│ [Undo all]  [Select imported]                             │
└───────────────────────────────────────────────────────────┘
```

- Banner style: `rounded-lg border border-green-200 bg-green-50 dark:border-green-900 dark:bg-green-950/30 px-4 py-3`
- `×` dismiss: sets `dismissed = true` via `PATCH /api/import-batches/[batchId]`
- **[Undo all]**: calls `DELETE /api/import-batches/[batchId]` which deletes all remaining cases (those still with `importBatchId`) in a single DB transaction. Cases already added to suites (importBatchId=null) are NOT deleted. Banner auto-dismisses when count reaches 0.

- Banner buttons: `flex flex-wrap gap-2` for mobile stacking
- **Banner shows live count** (computed from `_count: { cases: true }` on ImportBatch query, not static `caseCount`). Banner text: "47 cases imported from acme-tests.xlsx" → "37 cases remaining from acme-tests.xlsx" as cases join suites

## Feature 3: JIRA Key Format Enforcement

### Shared utility: `src/lib/jira.ts`
```ts
export function autoCorrectJiraKey(raw: string): {
  corrected: string | null;
  wasChanged: boolean;
  original: string;
} {
  const upper = raw.trim().toUpperCase();
  if (!upper) return { corrected: null, wasChanged: false, original: raw };
  // Already valid
  if (/^[A-Z][A-Z0-9]*-[0-9]+$/.test(upper)) {
    return { corrected: upper, wasChanged: upper !== raw.trim(), original: raw };
  }
  // Normalise: replace _ or space with -, strip other non-alphanumeric
  const normalised = upper.replace(/[_\s]/g, '-').replace(/[^A-Z0-9-]/g, '');
  const match = normalised.match(/^([A-Z][A-Z0-9]*)-?([0-9]+)$/);
  if (match) {
    return { corrected: `${match[1]}-${match[2]}`, wasChanged: true, original: raw };
  }
  return { corrected: null, wasChanged: false, original: raw };
}
```

Shared between import API (server) and form component (client).

Example corrections:
| Input | Corrected | Rule |
|-------|-----------|------|
| `ac-3` | `AC-3` | uppercase |
| `ac-03` | `AC-03` | uppercase |
| `AC03` | `AC-3` | insert hyphen |
| `PROJ123` | `PROJ-123` | insert hyphen |
| `PROJ_123` | `PROJ-123` | normalise underscore |
| `???` | null | cannot correct → skipped |
| `AC-003` | `AC-003` | valid as-is |

### Import flow change
Current: `select → map → import`
New: `select → map → [JIRA corrections, conditional] → import`

The JIRA corrections step only appears if the JIRA column is mapped AND at least one value is correctable or uncorrectable (non-empty, non-valid).

If all JIRA values are already valid or empty: skip the review step entirely.

### JIRA corrections review step UI

**Positive framing** (not an error screen):

```
Step 3 of 3 — Review JIRA key corrections

We can automatically fix some JIRA keys to match the required format (e.g. AC-03).

  Row  │ Original   │ Corrected to
  ─────┼────────────┼────────────────────
  2    │ ac-003     │ AC-003             ✓
  5    │ PROJ123    │ PROJ-123           ✓
  8    │ ac03       │ AC-3               ✓
  12   │ ???        │ (will be skipped)  —
  ─────┴────────────┴────────────────────
       4 will be corrected · 1 will be skipped

  [← Back to mapping]    [Apply corrections & import →]
```

- Corrected rows: `text-foreground` with green `✓`
- Skip rows: `text-muted-foreground/50` with `—` (neutral, not red)
- Footer: "N will be corrected · M will be skipped"
- All-skipped variant: "All JIRA keys could not be corrected and will be skipped. Go back to fix the source, or proceed without JIRA keys."

**Responsive:** On mobile (<480px), collapse to 2 columns (Original → Corrected). Row number moves to `title` attribute.

Updated `jiraKeySchema` in `test-cases/route.ts`:
```ts
const jiraKeySchema = z
  .string()
  .trim()
  .max(32)
  .optional()
  .nullable()
  .transform((val) => {
    if (!val) return null;
    const { corrected } = autoCorrectJiraKey(val);
    return corrected; // null if uncorrectable
  });
```

### Import preview endpoint (NEW — required for step 3)

`POST /api/test-cases/import/preview` — accepts the same multipart form as the import endpoint (file + mapping), parses the file, analyzes the JIRA column, and returns the correction analysis WITHOUT creating any records.

Response shape:
```ts
{
  jiraAnalysis: {
    needsReview: boolean;         // true if any corrections or skips exist
    rows: Array<{
      rowNumber: number;
      original: string;
      corrected: string | null;   // null = will be skipped
      wasChanged: boolean;
    }>;
    willCorrect: number;
    willSkip: number;
  }
}
```

**Note:** Preview uploads the file twice (once for preview, once for import). For large XLSX files this is redundant. Future optimization: the import step can re-use the already-known corrections rather than re-parsing. For v0.5.3, the simple approach is acceptable.


After "map" submission: call `/preview` → if `needsReview: true`, set `jiraAnalysis` and `importStep = "jira-review"`. Otherwise proceed directly to import.

### Page data (DECIDED: server-side fetch)
